# House style — how a review reads

> **Part of:** [pr-review](../SKILL.md). The analysis engines find issues; this file governs how the review is written. A review should read like a thoughtful senior engineer left it — and open a conversation, not deliver a verdict.

## Formatting rules

- **No severity badges.** No `CRITICAL`/`HIGH`/`MEDIUM`/`LOW`, no `major`/`minor`, no colored dots. If something blocks merge, say why in a sentence — that carries more than a label.
- **No emojis.**
- **No machine-generated footer.** No "generated by", no rate-this-comment line. This is a human review posted by the user.
- **No performative praise.** Skip "Great work!" and "Nice PR!". Reinforcing something genuinely well done is fine — state it plainly.
- **Plain citations.** Reference `path/to/file.py:42` or a normal permalink; don't manufacture synthetic anchors.
- **No numbered lists for findings.** Use prose or `-` bullets, so the user can drop an item in the approval step without renumbering the rest.

## Voice

- Talk to the author, not about them: "This retries forever if the host is down — want a cap?" beats "The retry loop lacks a bound."
- Lead with the problem and its consequence, then offer the fix as a suggestion. The author owns the code.
- Be specific and short. One concrete sentence about what breaks and when beats a paragraph of hedging.
- Show uncertainty when you have it: "I might be missing context, but…" invites the discussion that's the point of a review.
- Match the repo's register — mirror the tone of the existing human comments rather than importing a house accent.

## Pushback and discussion

A review starts a conversation; it isn't a one-way verdict. This matters most when an automated reviewer was involved — the goal is human judgment steering the outcome, not two agents agreeing with each other into questionable output.

- When you engage an existing thread, react to it: agree and add, build on it, or disagree with reasoning. Don't silently re-raise a point someone already made.
- When the author likely had a reason you can't see, ask rather than assert.
- Evaluate every suggestion on its merits, whoever (or whatever) raised it. It's fine for a review to say plainly that an earlier suggestion — from a person or a bot — doesn't hold up, and why.
- The user sets the final tone and may soften or sharpen any finding in the approval step.

## The summary body

A short paragraph or two: what the PR does, your overall read (close, or real blockers?), and the headline concerns. The inline comments carry the specifics. Mention test coverage only when it's genuinely thin on changed behavior, not as a reflex.

## Examples

These illustrate the voice — they aren't a checklist of specific issues to look for.

**Inline finding — avoid (report style):**

> ### HIGH | `src/worker.ts:142` | Unbounded retry
> 🔴 The retry loop has no maximum attempt count, which could cause a thundering herd. You should add exponential backoff.

**Inline finding — prefer (house voice):**

> This retry loop has no ceiling — if the downstream is down it spins until the request times out, and every worker doing the same turns an outage into a thundering herd. Worth a max-attempts cap with backoff.

**Engaging an existing thread:**

> Building on @alice's note above — beyond the naming, this also shadows the outer `config`, so the env override silently does nothing. Renaming fixes both.

**Disagreeing with a suggestion (from a human or a bot):**

> An earlier comment suggested extracting this constant, but the value is the documented limit from the spec linked just above; inlining it with that link reads clearer than a named constant that only points back here. I'd leave it.

**Summary body:**

> Adds the jobs API and wires it to the existing queue. Structure is clean and the happy-path tests are solid. Two things I'd want resolved before merge: the retry has no ceiling (inline), and the new handler trusts the `status` query param without validation (inline). The rest are small.
